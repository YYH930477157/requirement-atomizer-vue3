# 文档量子化技术方案

专家评审版

## 1. 背景与目标

本方案面向能源计量行业，特别是电表、DLMS/COSEM、OBIS、xDLMS、SAP、HLS/LLS、COSEM interface class 等技术标准和产品需求文档。

这里的“文档量子化”不是简单分段，而是将技术标准文档拆解为最小、可追溯、可验证、可由 AI 和专家共同复核的结构化需求单元。

目标是建立一条可落地的需求解析流水线：

```text
原始 Word 标准文档
-> 文档结构块
-> 表格语义结构
-> 行业知识库匹配
-> 原子需求候选
-> 大模型复核任务
-> 专家可审查需求清单
```

该方案的核心原则是：

- 规则引擎负责高覆盖、可追溯、稳定的初步量子化。
- 大模型负责语义复核、纠错、合并、补漏。
- 专家负责最终确认高风险和标准解释类内容。

## 2. 适用文档范围

当前方案主要适用于 `.docx` 技术标准文档，尤其适合以下类型：

- 电表技术规范
- DLMS/COSEM 标准或 profile 文档
- ABNT、IEC、INMETRO 等标准文档
- COSEM 对象表
- COSEM 属性访问权限表
- xDLMS 服务矩阵
- 安全策略表
- 安全套件表
- 事件编号表
- 计量量与单位表
- profile status、flag、error、alarm 定义表

当前验证样本文档：

```text
Appendix 9-ABNT NBR 16968-2022 EN.docx
```

## 3. 总体架构

系统分为五层：

```text
文档解析层
-> 表格语义增强层
-> 行业知识库匹配层
-> 原子需求生成层
-> AI/专家复核层
```

### 3.1 文档解析层

负责从 Word 文档中提取：

- 标题
- 段落
- 表格
- 顺序编号
- 章节路径
- 文档区域，例如正文、前言、目录等

输出文件：

```text
blocks.jsonl
```

### 3.2 表格语义增强层

负责处理标准文档中大量复杂表格。

当前支持：

- 多行表头识别
- 重复表头去重
- 表格行结构化
- `X` 标记矩阵解析
- 非 `X` 值矩阵解析
- COSEM 对象上下文继承
- RC/PC/SC/LC 访问权限拆解

输出文件：

```text
table_items.jsonl
```

### 3.3 行业知识库匹配层

知识库以 Obsidian Markdown 作为人工维护源，以 JSON 作为运行时格式。

```text
Obsidian Markdown Vault
-> compiled JSON KB
-> Python API / CLI / HTTP API
-> Atomizer / Agent / Windows App
```

知识库用于识别：

- DLMS
- COSEM
- OBIS
- IEC 62056
- xDLMS services
- COSEM class
- SAP
- Association
- HLS / LLS
- Security Suite
- Access rights RC/PC/SC/LC
- Load profile
- Billing profile
- Event log
- Alarm / Error

### 3.4 原子需求生成层

规则引擎基于文档结构、表格语义和知识库匹配结果，生成可复核的候选原子需求。

输出文件：

```text
atomic_requirements.jsonl
```

### 3.5 AI/专家复核层

系统同时生成大模型复核任务。

输出文件：

```text
llm_tasks.jsonl
```

推荐流程：

```text
规则生成候选
-> 大模型复核、合并、补漏
-> 专家审核高风险项
-> 最终需求库
```

## 4. 核心输出文件

### 4.1 blocks.jsonl

文档块级输出。

包含：

- block_id
- 顺序号
- 类型：heading、paragraph、table
- 文本
- 章节路径
- 领域标签
- 知识库匹配
- 是否疑似需求

### 4.2 chunks.jsonl

面向 RAG 和大模型的上下文块。

用途：

- 大模型复核
- 语义补漏
- 上下文检索
- 需求解释

### 4.3 table_items.jsonl

表格行级结构化输出。

对于普通表格，输出字段：

```json
{
  "item_id": "TBL-000007-R000003",
  "table_title": "Table 7 - Events",
  "row_index": 3,
  "fields": {
    "Group number": "1",
    "Subgroup number": "10",
    "Event number": "1",
    "Description of the event": "Reboot with data loss"
  }
}
```

对于矩阵表，保留 matrix_facts：

```json
{
  "subject": "Public customer",
  "predicate_header": "xDLMS Service / GET",
  "marker": "X",
  "relation": "allowed"
}
```

对于 COSEM 对象表，保留对象上下文：

```json
{
  "fields": {
    "#": "1",
    "Object/attribute name": "logical_name",
    "Access rights RC/PC/SC/LC": "R-/R-/R-/R-"
  },
  "cosem_object_context": {
    "object_name": "SAP Assignment",
    "class_id": 17,
    "obis": "0-0:41.0.0.255"
  }
}
```

### 4.4 atomic_requirements.jsonl

规则引擎生成的候选原子需求。

示例：

```json
{
  "req_id": "AREQ-000063",
  "source_type": "cosem_attribute_row",
  "source_refs": ["BLK-000608", "TBL-000031-R000003"],
  "domain": "cosem_object",
  "object": "SAP Assignment.logical_name",
  "requirement_type": "cosem_attribute_access",
  "requirement": "COSEM attribute SAP Assignment.logical_name for SAP Assignment / CL 17 / OBIS 0-0:41.0.0.255 shall use access rights R-/R-/R-/R-.",
  "verification_method": "configuration_check",
  "ambiguity": false,
  "confidence": 0.9
}
```

### 4.5 llm_tasks.jsonl

大模型复核任务。

用途：

- 纠错
- 合并重复需求
- 拆分过粗需求
- 补充遗漏需求
- 标记歧义和专家问题

### 4.6 quality_report.json

质量统计报告。

包含：

- 原子需求总数
- 类型分布
- 来源分布
- 低置信需求数量
- 歧义需求数量
- 表格候选覆盖率
- 领域表格候选覆盖率

示例：

```json
{
  "counts": {
    "atomic_requirements": 2337,
    "ambiguous_atomic_requirements": 6,
    "low_confidence_atomic_requirements": 83
  },
  "coverage": {
    "body_table_candidate_ratio": 0.9928,
    "domain_table_candidate_ratio": 0.9966
  }
}
```

## 5. 当前支持的原子需求类型

当前已支持以下 requirement_type：

```text
paragraph requirement
capability_matrix
association_security_matrix
table_value_matrix
cosem_object_instance
cosem_attribute_access
event_definition
event_group_retention
security_suite_definition
security_policy_bit
security_policy_state
measurement_quantity_unit
flag_definition
```

### 5.1 capability_matrix

用于 xDLMS 服务矩阵。

示例：

```text
Public customer shall support xDLMS Service: GET.
```

### 5.2 association_security_matrix

用于关联矩阵中的安全级别。

示例：

```text
Local and remote clients management and measurement shall have Management Logical Device set to High Level of Security (HLS).
```

### 5.3 cosem_object_instance

用于 COSEM 对象实例定义。

示例：

```text
COSEM object SAP Assignment / CL 17 / OBIS 0-0:41.0.0.255 shall be defined by the profile.
```

### 5.4 cosem_attribute_access

用于 COSEM 属性访问权限。

示例：

```text
COSEM attribute SAP Assignment.logical_name for SAP Assignment / CL 17 / OBIS 0-0:41.0.0.255 shall use access rights R-/R-/R-/R-.
```

并拆解为：

```json
{
  "access_rights_by_client": {
    "clients": [
      {"client": "RC", "code": "R-", "read": true, "write": false, "allowed": true},
      {"client": "PC", "code": "R-", "read": true, "write": false, "allowed": true},
      {"client": "SC", "code": "R-", "read": true, "write": false, "allowed": true},
      {"client": "LC", "code": "R-", "read": true, "write": false, "allowed": true}
    ]
  }
}
```

### 5.5 event_definition

用于事件编号定义。

示例：

```text
Event G1-SG10-E1 shall be defined as: Reboot with data loss.
```

### 5.6 event_group_retention

用于事件组最小记录数。

示例：

```text
Event subgroup G1-SG10 shall keep at least 100 records for Non-special events.
```

### 5.7 security_suite_definition

用于安全套件定义。

示例：

```text
Security suite 1 shall be defined as ECDH-ECDSA-AES-GCM-128-SHA-256.
```

### 5.8 security_policy_bit

用于安全策略 bit 定义。

示例：

```text
Security policy bit 0 shall be defined as: Not used, must be set to "0".
```

### 5.9 security_policy_state

用于安全策略 state 定义。

示例：

```text
Security policy state 1 shall be defined as: All messages are authenticated.
```

### 5.10 measurement_quantity_unit

用于计量量和单位。

示例：

```text
Measurement quantity Total incremental direct active energy shall use unit Wh.
```

### 5.11 flag_definition

用于状态码、flag、profile status 定义。

示例：

```text
Profile status Bit 7 IV shall be defined as: Invalid data.
```

## 7. 专家评审重点

建议专家重点评审以下问题。

### 7.1 需求化边界

需要确认：

- 表格定义项是否都应转换为 requirement。
- `shall be defined as` 是否适合作为统一表达。
- 安全套件、事件编号、计量单位更适合叫 requirement 还是 specification item。

### 7.2 DLMS/COSEM 语义准确性

需要确认：

- RC/PC/SC/LC 的客户端解释是否符合该文档上下文。
- COSEM object instance 的 CL 和 OBIS 解析是否完整准确。
- Association LN、SAP Assignment、Security Setup 等对象是否应进入最终需求库。

### 7.3 事件表处理

需要确认：

- `Group number = 1, 2, 3, 4, 5, 6` 是否应展开为多条事件。
- `Subgroup = All` 是否应保持原文还是展开。
- 事件定义是否需要附带事件组和日志容量要求。

### 7.4 规则置信度

需要确认：

- 哪些 requirement_type 可自动接受。
- 哪些 requirement_type 必须专家复核。
- 低置信项阈值是否合理。

### 7.5 大模型角色

需要确认大模型应执行哪些任务：

- 重写自然语言
- 合并重复需求
- 发现遗漏需求
- 判断定义项是否为需求
- 生成测试点
- 生成审查问题

## 8. 当前限制

当前方案仍存在以下限制：

- 仅支持 `.docx`。
- 复杂跨页表格依赖 Word 文档本身结构。
- 机器翻译质量会影响段落类需求判断。
- 表格定义项是否应需求化需要行业专家确认。
- 复合 group/subgroup/event 编号暂未自动展开。
- 图片、公式、脚注尚未深度解析。
- 规则候选仍需大模型和专家复核。

## 10. 结论

当前方案已经从简单文档切块升级为面向能源计量标准的需求量子化引擎。

它的核心价值是：

- 保留文档可追溯性。
- 能理解大量标准表格。
- 能结合行业知识库。
- 能先生成高覆盖原子需求草稿。
- 能为大模型和专家提供可复核输入。

总体定位：

```text
规则引擎负责稳定、高覆盖、可追溯；
大模型负责语义复核、纠错、补漏；
专家负责最终判定和标准解释。
```
